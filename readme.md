# DST-Nucleus

A zero-dependency Don't Starve Together server core.

## Core Design Philosophy

* **Single Binary <10M**
* **Flat Memory Profile**
* **Zero Database**
* **Zero Dependencies**
* **Zero OOP Bloat**

## Key Features

### Automated Updates
* Periodically checks for game and mod updates. If an update is detected, it initiates a graceful shutdown and upgrade sequence.
* Broadcasts in-game announcements notifying players of a forced shutdown in X seconds, repeating every 5 seconds. If all players disconnect during this period, the server shuts down and updates immediately.
* Utilizes native DST commands, allowing the game server to autonomously decide whether to broadcast announcements or shut down based on the current player count.
* Automatically detects and updates enabled mods. No need to maintain a separate update list; all active mods are automatically appended. You can still maintain a custom list to update disabled mods by modifying `dedicated_server_mods_setup.lua`. The engine will auto-detect and append any active mods missing from this file.

### Crash Recovery
* Strictly adheres to the **"Let it Crash"** principle. Crash = Shutdown. Across all operational logic, intentional shutdowns and unexpected crashes are treated identically: it just restarts.
* A crash in either world triggers a synchronous shutdown of the other, binding the two DST worlds in a state of absolute shared fate.
* Naturally, configuring it to run only the Master world is fully supported without issues.

### Dual-Server Architecture
* Even if your two worlds are hosted on completely separate physical machines, or even in different countries, the synchronized crash-and-restart mechanism remains fully effective.
* You can enforce asymmetric dependency: if the Master crashes, the Caves restart with it; but if the Caves crash, they restart alone while the Master remains unaffected.
* Cross-server restart synchronization relies on mutual API polling to detect epoch mutations. It tolerates any level of network instability.
* No matter the time gap between restarts, or how long the network connection was severed, as long as a restart occurred, a single successful API ping is enough for the peer to detect the state mutation and initiate a cascading restart.

### Powerful API [Document](api.md)
* Exposes a comprehensive set of API endpoints.
* Query real-time game status.
* Dispatch custom Lua commands directly to a specified server.
* **Sub-millisecond real-time log streaming:** Catch the raw STDOUT waterfall via SSE. By decoupling command injections (Input) and log capturing (Output), you hold the ultimate I/O gateway to build absolutely anything.
* Start, stop, and restart servers—delivering a `systemctl`-like bare-metal experience.
* Read and write configuration files with 1:1 raw transmission. What you see is what you get. Paste your configs directly and write them; the API makes zero attempts to mutate your data.
* Includes a minimalist UI covering essential functions: reading/writing configs, toggling server states, and monitoring game status.
* **Extremely lightweight execution:** Aside from a few heap allocations for memory safety, the vast majority of APIs achieve **zero-allocation** and **zero-copy**.

### In-Game State Probing
* Operates entirely mod-free. Probing commands are injected parasitically into the DST process directly via the console. *(Note: This feature currently lacks extensive testing, has only passed basic checks, and data accuracy is not yet fully guaranteed).*
* I entirely reject bloated JSON or string serialization. DST and the Core communicate via raw binary payloads injected directly into standard I/O pipes. By relying exclusively on pure bitwise operations, I slash IPC overhead by 50%-90%.
* Tracks days survived, player count, current season, days until season change, world temperature, rain/snow states, time of day, moon phases, and ruins nightmare cycles.
* Tracks the status of Epic Bosses: Deerclops, Bearger, Moose/Goose, Dragonfly, Bee Queen, Klaus, Toadstool, Ancient Fuelweaver, and Antlion's enrage timer.

## Deployment (Linux Systemd)
Assuming you are using the standard `steam` user for your DST dedicated server:

### 1. Download & Extract
Fetch the latest pre-compiled binary, extract it, and clean up the tarball.

```
cd /home/steam

wget https://github.com/njj0121//dst-nucleus/releases/latest/download/dstn-linux-amd64.tar.gz
tar -xzf dstn-linux-amd64.tar.gz
rm dstn-linux-amd64.tar.gz
```

### 2. Set Permissions
Lock down the directory. The binary must be executable and owned by the steam user.

```
chown steam:steam /home/steam/dstn
chmod +x /home/steam/dstn
```

### 3. Generate Configuration
Run the binary. If no configuration file is found, it will automatically generate a fully commented config.yaml in the current working directory.<br>
The -c flag dictates both loading and generation—if the specified target path doesn't exist, a fresh config will be forged there. You can then tweak it to fit your setup.<br>
You can also review [default.yaml](default.yaml) for a direct look at the auto-generated output.

```
cd /home/steam
./dstn
chown steam:steam /home/steam/config.yaml
chmod 666 /home/steam/config.yaml

#./dstn -c config.yaml
#./dstn -c /home/steam/config.yaml
```

### 4. Setup Systemd Service
Inject the service configuration directly into systemd using a single cat command. This ensures the daemon runs cleanly in the background under the isolated steam user. (Requires sudo privileges).

```
sudo bash -c 'cat > /etc/systemd/system/dstn.service << EOF
[Unit]
Description=DST Nucleus
After=network.target

[Service]
Type=simple
User=steam
Group=steam
WorkingDirectory=/home/steam
ExecStart=/home/steam/dstn
Restart=on-failure
RestartSec=5s

[Install]
WantedBy=multi-user.target
EOF'
```

### 5. Enable & Start
Reload the daemon and ignite the core.

```
sudo systemctl daemon-reload
sudo systemctl enable --now dstn.service
```

## Deployment (Windows)
### 1. Download & Extract
Open CMD  and run these commands line by line. We use native Windows tools—no third-party bloat required.
```
mkdir C:\dstn
cd C:\dstn
curl -f -L -O https://github.com/njj0121/dst-nucleus/releases/latest/download/dstn-windows-amd64.zip
tar -xf dstn-windows-amd64.zip
del dstn-windows-amd64.zip
```

### 2. Generate Configuration
Run the binary. If no configuration file is found, it will automatically generate a fully commented config.yaml in the current working directory.<br>
The -c flag dictates both loading and generation—if the specified target path doesn't exist, a fresh config will be forged there. You can then tweak it to fit your setup.<br>
You can also review [default.yaml](default.yaml) for a direct look at the auto-generated output.

```
.\dstn.exe
:: .\dstn.exe -c config.yaml
:: .\dstn.exe -c "C:\dstn\config.yaml"
```

### 3. Run
Once your configuration is set, run the command again. Leave the terminal window open to monitor the raw IPC stream and system logs in real-time.

```
.\dstn.exe
```

## Antivirus False-Positive Disclaimer
By design, this engine performs highly aggressive OS-level operations: cross-process state probing, continuous Lua payload injection, and ruthless termination of zombie processes. These mechanics will inevitably trigger the dynamic behavioral analysis (heuristics) of certain Antivirus software, flagging it as a potential Trojan during runtime.

The Physical Truth: > The binary contains zero malicious code. A static deep scan of the executable (physically verified clean via Microsoft Defender) will prove its safety.

If you are paranoid about the pre-compiled binary:

Perform a deep static scan on the file yourself.

If you still do not trust it, audit the Go source code and compile it yourself using `go build`.

**Do NOT open issues regarding automated AV behavioral blocks.**
